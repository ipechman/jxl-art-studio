📖 Cheat sheet — how tree code works

The image is generated by a tree of if questions, answered once for every pixel of every color channel (channel 0=red, 1=green, 2=blue). Leaves choose a predictor plus an offset — and that becomes the pixel value. There is no error correction: the prediction is the image. 

Width 512 Height 512
if x > 256
 - Set 255      ← right half: white
 - N 2          ← left half: pixel above + 2

Header options (all optional)

Width Height · Bitdepth 8 (values 0–255; 16 → 0–65535) · RCT 6 (color transform 0–41, changes the palette) · Upsample 2|4|8 · Orientation 0–7 (rotate/mirror) · GroupShift 0–3 (block size for g) · Alpha · Squeeze · Spline … EndSpline (smooth glowing curves)

Things you can ask about (properties)

c channel · g block number · x y position · N pixel above · W pixel left · |N| |W| absolute values · neighbor differences W-NW, NW-N, N-NE, N-NN, W-WW, W+N-NW · WGH · Prev (same pixel, previous channel)

Ways to paint a pixel (predictors)

- Set 200 exact value · - W 4 left + 4 · - N -2 above − 2 · - NW - NE - WW · - Gradient smooth plane · - AvgW+N (and other Avg pairs) · - AvgAll · - Select · - Weighted self-correcting (chaotic & fun)

Good to know

  • Only > comparisons exist. Both branches are required: first the “yes” branch, then the “no” branch.
  • Line breaks don’t matter — only word order does. /* comments */ need spaces around them.
  • On the first row/column, missing neighbors fall back: N→W, W→N, corners→0.
  • Values outside the bit depth are kept while computing and only clipped for display — overflow is an art technique!
